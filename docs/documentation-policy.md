# 문서 운영 정책

## 목적
개발 진행 상황과 코드 상태가 문서에 즉시 반영되도록, 문서 작업을 개발 작업과 같은 우선순위로 관리합니다.

## 기본 원칙
- 기능/버그 수정/리팩터링 작업을 수행할 때 **관련 문서도 같은 작업 단위에서 함께 수정**합니다.
- 신규 기능이 추가되면 최소한 아래 항목을 갱신합니다.
  - 사용자 실행/검증 방법
  - 영향 받는 화면(UI) 또는 시스템 동작
  - 테스트/검증 스크립트 사용 방법
- 문서의 기본 작성 언어는 **한국어**로 유지합니다.

## 커밋/머지/PR 작성 규칙
- 커밋 메시지, PR 제목/본문, 머지 커밋 메시지는 가능한 한 **한국어**로 작성합니다.
- 커밋 단위는 “코드 변경 + 관련 문서 변경”이 함께 리뷰 가능하도록 구성합니다.
- 문서만 수정한 경우에도 변경 목적과 범위를 커밋 메시지에 명확히 기록합니다.

## 권장 체크리스트
작업을 마무리하기 전에 아래를 확인합니다.

- [ ] README 또는 관련 docs 파일에 최신 동작이 반영되었는가?
- [ ] 실행/검증 명령이 현재 코드 기준으로 유효한가?
- [ ] 화면 구조나 용어가 실제 UI와 일치하는가?
- [ ] 커밋/PR/머지 메시지를 한국어로 작성했는가?

## 문서 업데이트 우선순위
1. `README.md` (실행/검증/운영에 직접 필요한 정보)
2. `docs/project-overview.md` (구조/아키텍처 개요)
3. 기능별 상세 문서 (예: QA, 밸런스, 작업 이력)
